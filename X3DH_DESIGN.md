# X3DH — дизайн-документ (проектирование ДО кода)

> Живой документ. Развилки проектируются поблочно, фиксируются по мере
> закрытия. Код пишется ТОЛЬКО по закрытой развилке. Крипто-ошибка =
> уязвимость, не баг. Путь 1 (свой X3DH на P-256, см. CRYPTO_ROADMAP.md).

## МОДЕЛЬ УГРОЗ (честная, без ложных обещаний)
Что защищаем и чем:
- ЛИЧНОСТЬ (нельзя выдать себя за пользователя) — Sign-ключ в KeyStore/TEE.
  Несокрушимо даже при Root: приватный ключ неизвлекаем из TEE.
- ИСТОРИЯ переписки при компрометации — Forward Secrecy (эфемерные ключи
  X3DH, позже Double Ratchet). Украл ключ сегодня — вчерашнее не расшифруешь.
- ФАЙЛ ключей от кражи (не-Root) — шифрование хранилища (SQLCipher).
Что НЕ защищаем (честно):
- DH-ключ от активного Root — атакующий с Root читает память в момент
  использования. Принимаемое ограничение. Защита истории = FS, не хранение.
- Root-детект — это ДАТЧИК для пользователя ("возможна компрометация"),
  НЕ элемент модели безопасности. Обходится тем, кто уже получил Root.
  Не входит в расчётную модель. Не обещаем "защищённость при Root".

## РАЗВИЛКА 1: подпись Signed Prekey — ЗАФИКСИРОВАНО ✅

### Модель Identity: ПАРА ключей (Вариант A)
- Identity_Sign — в KeyStore/TEE (PURPOSE_SIGN). Подписывает SPK.
  Приватная часть неизвлекаема. Из него же stableId (SHA256 pub).
- Identity_DH — в OpenSSL, приватная часть в шифрованном хранилище.
  Участвует в ECDH-операциях X3DH (DH2 = EK_A ↔ IK_B требует ECDH от IK).
- Причина пары: KeyStore-ключ PURPOSE_SIGN не может делать ECDH,
  а X3DH требует от identity И подпись, И DH -> два ключа.

### Объект подписи (Вариант Б)
Подписываем конкатенацию: SPK_pub ‖ timestamp ‖ key_id.
Причина: позволяет РОТАЦИЮ SPK (новый timestamp инвалидирует старый),
защита от подсовывания устаревшего prekey. (Не только SPK_pub — вариант А
сломался бы при первой ротации.)

### Алгоритм и форматы (совместимость проверена по коду)
- Подпись: Signature "SHA256withECDSA" в KeyStore -> DER (ASN.1).
- Проверка: OpenSSL EVP_DigestVerify -> ест DER НАТИВНО. Конверсии НЕТ.
- Публичный ключ Android: publicKey.encoded = X.509 SubjectPublicKeyInfo.
  OpenSSL d2i_PUBKEY (УЖЕ используется в KeyExchange) читает этот формат.
  Обе стороны на X.509 DER -> совместимы из коробки.

### Модель доверия: TOFU + fingerprint
Trust On First Use: доверяем ключу при первом контакте.
Слабость (честно): уязвим к MITM в момент ПЕРВОГО контакта.
Митигация: ручная сверка fingerprint (get_fingerprint УЖЕ есть) —
сравнить код лично/по видео для параноидального режима.

### Осталось реализовать по Развилке 1 (код, отдельные кирпичи)
- signData(data) в KeyStoreManager (подпись через IDENTITY_ALIAS -> DER)
- getIdentityPublicKeyEncoded() -> X.509 DER
- проверка подписи в OpenSSL (EVP_DigestVerify) — C++ сторона
- ТЕСТ совместимости: Android подписал -> OpenSSL проверил (кирпич-проверка)

## РАЗВИЛКА 2: хранение ключей (Storage Layer) — ЗАФИКСИРОВАНО
Факт установлен: мастер-пароля НЕТ. Ключ БД автогенерится (SecureRandom),
шифруется AES-GCM ключом из KeyStore (libcryptsafe_db_key), passphrase
хранится в prefs (enc_passphrase). Argon2ID НЕ нужен (нет польз. пароля).

### ЧЕСТНАЯ ПЕРЕОЦЕНКА (отсеян security theater)
- Затирание памяти fill(0) — ГИГИЕНА, не барьер. JVM/GC копирует объекты,
  контроля над памятью нет. Против Root не держит. Оставляем как хороший тон.
- Защита памяти от чтения на JVM без Root — против Root невозможна.
  НЕ строим на этом уверенность (тот же урок, что Root-детект).
- is_extractable=false — УЖЕ выполнено фактом AndroidKeyStore (дефолт).
  Ключ неизвлекаем из TEE по определению.

### РЕАЛЬНЫЕ ЗАЩИТЫ (аппаратные + математические)
1. TEE-привязка ключа БД (KeyStore) — ключ неизвлекаем, клон устройства
   невозможен без физического чипа.
2. Forward Secrecy (X3DH эфемерные ключи) — ГЛАВНАЯ защита истории при
   компрометации. Украл ключ сегодня -> вчерашнее не расшифровать.
3. AES-256-GCM (БД) + X3DH (сессии) — крипто-стек.

### СТАТУС StrongBox (обновлено, коммит 38f7510)
- ПОДТВЕРЖДЕНО ФАКТОМ на первом телефоне (30a10da0): лог показал
  "ключ БД создан в StrongBox" при холодном старте с чистым ключом.
- Второй телефон (b7549a6f): тот же чип (feature=300), тот же код (APK).
  Прямой тест невозможен — pm clear заблокирован защитой приложения
  (SecurityException CLEAR_APP_USER_DATA — приложение защищено, это ОК).
  Применится идентично (детерминировано: код+чип те же). Не пробел.
- Нюанс теста: StrongBox логируется только при ХОЛОДНОМ старте с чистым
  ключом. force-stop + pm clear (или снос через приложение) -> запуск.
  Тёплый старт (bring to top) ключ не пересоздаёт.

### УСИЛЕНИЕ: StrongBox с fallback (первый кирпич кода Р2)
- setIsStrongBoxBacked(true), catch StrongBoxUnavailableException -> TEE.
- StrongBox = выделенный чип, изоляция даже при kernel exploit.
- ФАКТ: первый телефон (30a10da0) StrongBox ПОДДЕРЖИВАЕТ (feature=300).
- СТРАТЕГИЯ: "новый ключ для новых" (low friction): ключ есть -> как есть
  (старые на TEE, тоже надёжно); ключа нет -> StrongBox+fallback.
  БЕЗ миграции/перешифровки (не ломаем существующие БД).

### ОТЛОЖЕНО (UX-развилка)
- setUserAuthenticationRequired (биометрия на каждый доступ к ключу БД):
  компромисс паранойя vs удобство. Решить после определения App-Lock.

## РАЗВИЛКА 3: KDF (деривация сессионного ключа) — ЗАФИКСИРОВАНО
Факт: сейчас в ядре SHA256(secret) (строка 66 KeyExchange.hpp) — антипаттерн
для X3DH. HKDF доступен в OpenSSL билде (kdf.h, EVP_PKEY_HKDF). Меняем на HKDF.

### АЛГОРИТМ: HKDF-SHA256 (RFC 5869), стандарт Signal
Двухэтапный Extract-then-Expand:
- Extract: HKDF-Extract(salt, IKM) -> Master Secret (MS)
- Expand: HKDF-Expand(MS, info, len) -> рабочие ключи

### ВХОД (IKM) — с префиксом доменного разделения
IKM = F ‖ DH1 ‖ DH2 ‖ DH3 ‖ DH4
- F = 0xFF × 32 (доменное разделение, как в спеке X3DH). НЕ пропускать:
  гарантирует несовпадение входа с другими протоколами. Стандарт = аудируемо.
- Порядок DH СТРОГИЙ: DH1=IK_A↔SPK_B, DH2=EK_A↔IK_B,
  DH3=EK_A↔SPK_B, DH4=EK_A↔OPK_B. Перепутать порядок -> разные ключи у сторон.
- Каждый DH = 32 байта (P-256 ECDH через существующий compute_shared_key).

### SALT: ФИКСИРОВАННЫЙ (не случайный)
- salt = нули (0x00 × 32) или константа протокола.
- Причина: IKM уже высокоэнтропийный (4 DH). Уникальность сессии даёт
  ЭФЕМЕРНЫЙ ключ Алисы (новый каждый раз -> DH разные -> MS разный).
- Случайную соль в пакет НЕ кладём: в асинхронном X3DH (Боб офлайн) её
  негде согласовать + лишние байты (traffic analysis). Забота о минимизации
  метаданных верна, но применяется в Р5 (дизайн пакета), не здесь.

### EXPAND: раздельные ключи по info-меткам
Kenc = HKDF-Expand(MS, "LibCryptSafe_v1_AES_GCM_Key", 32)
Kauth = HKDF-Expand(MS, "LibCryptSafe_v1_HMAC_Key", 32)
Раздельные метки -> ключи математически не связаны.

### ХРАНЕНИЕ: производные ключи, НЕ вечный MS
- Храним Kenc/Kauth (или ratchet-состояние), НЕ Master Secret навсегда.
- Причина: Forward Secrecy. Хранишь MS вечно -> взлом БД -> вся история
  расшифрована -> FS сломана. MS используется и уничтожается.
- ЧЕСТНОЕ ОГРАНИЧЕНИЕ X3DH-без-ratchet: FS на уровне СЕССИИ (новая
  сессия = новый эфемерный = новый ключ), НЕ на уровне сообщения.
  Полная per-message FS придёт с DOUBLE RATCHET (следующий слой).
- Батарея: X3DH выполняется 1 раз при установке сессии; дальше Double
  Ratchet проворачивает ключи дёшево (KDF-цепочка, без новых DH). Забота
  о ресурсах решается ratchet, не хранением якоря.

### СВЯЗЬ С БУДУЩИМ (Double Ratchet)
MS/производные -> корень ratchet-цепочки. X3DH даёт стартовый секрет,
Double Ratchet обеспечивает per-message FS + self-healing. Отдельный слой.

## РАЗВИЛКА 4: последовательность X3DH-обмена — ЗАФИКСИРОВАНО
Эталон Signal X3DH, адаптирован под P-256 ядро.

### ПОСЛЕДОВАТЕЛЬНОСТЬ
1. Боб публикует на relay связку: IK_pub, SPK_pub+подпись, пачка [OPK_pub].
   Может затем уйти офлайн.
2. Алиса запрашивает связку Боба с relay (даже если Боб офлайн).
   Relay отдаёт IK, SPK+подпись, ОДИН OPK (и удаляет его из пачки).
3. Алиса ПРОВЕРЯЕТ подпись SPK (ECDSA, IK_Sign Боба). Защита от MITM.
   Генерит эфемерный EK_A.
4. 4 DH-операции (P-256, compute_shared_key), СТРОГИЙ порядок:
   DH1=IK_A↔SPK_B, DH2=EK_A↔IK_B, DH3=EK_A↔SPK_B, DH4=EK_A↔OPK_B.
5. SK = HKDF(F ‖ DH1‖DH2‖DH3‖DH4). Общий ключ.
6. Первое сообщение Алисы -> Боб: EK_A_pub, opk_id, шифртекст.
   Боб (вернувшись) повторяет те же DH -> тот же SK.

### Р4.1: Identity в DH — DH-компонент (не Sign)
IK = пара (Развилка 1). В DH-операциях (DH1, DH2) участвует IK_DH
(OpenSSL DH-ключ). IK_Sign (KeyStore) только подписывает SPK.
DH1(IK_A) и DH2(IK_B) дают взаимную аутентификацию. Разделение позволяет
менять алгоритм подписи независимо от DH.

### Р4.2: исчерпание OPK — graceful degradation (DH1-DH3)
- Relay вернул OPK -> полный X3DH (DH1-DH4), в сообщении есть opk_id.
- Relay вернул ERR_NO_OPK -> X3DH без DH4 (DH1-DH3), opk_id отсутствует.
- КРИТИЧНО (устранение неоднозначности): наличие DH4 определяется НЕ флагом
  внутри IKM, а полем opk_id в ЗАГОЛОВКЕ первого сообщения. Каждый DH=32б
  фиксированно. Боб из opk_id знает: собирать IKM с DH4 или без. Флаг внутри
  IKM НЕ кладём (иначе неоднозначность длины -> потенциальная коллизия).
- Ослабление: без DH4 нет one-time защиты, но IK+EK+SPK остаются. Приемлемо.
- ИЗВЕСТНОЕ ОГРАНИЧЕНИЕ (relay полу-доверен): relay может соврать
  "OPK кончились" (downgrade на DH1-DH3) или переиспользовать OPK.
  Полностью не предотвратить без доверия к relay. Как в Signal.
  Митигация: Боб регулярно пополняет пачку OPK.

### Р4.3: хранение и жизненный цикл prekeys
- Боб хранит в SQLCipher приватные части: SPK_priv (долгоживущий),
  OPK_priv (пачка одноразовых, по id).
- Синхронизация использования OPK (Боб может быть офлайн при запросе):
  relay удаляет выданный OPK_pub сразу (не выдать дважды);
  Боб удаляет OPK_priv когда ПОЛУЧИТ и обработает первое сообщение с этим
  opk_id. Пока офлайн — OPK_priv ждёт в БД.
- Удаление OPK_priv после использования = залог Forward Secrecy
  (украли телефон завтра -> старое не расшифровать, ключ стёрт).

## РАЗВИЛКА 5.2: prekeys на relay (схема + логика) — ЗАФИКСИРОВАНО
Принцип архитектурного реализма: relay ВИДИТ граф (кто чьи prekeys берёт) —
не скрыть на централизованном узле (иначе не найдёт ключи). Скрываем
СОДЕРЖИМОЕ (не ключи — они публичны) и не ХРАНИМ логи запросов на диске.
Полное сокрытие графа (onion/PIR/multi-relay) = будущая веха (RETICULUM).

### Таблица prekeys (SQLite, по образцу queue из K6)
CREATE TABLE prekeys (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  recipient TEXT NOT NULL,     -- senderId Боба (владельца)
  key_type TEXT NOT NULL,      -- 'IK' | 'SPK' | 'OPK'
  key_id INTEGER NOT NULL,     -- id ключа (OPK: Боб найдёт priv по нему)
  key_value TEXT NOT NULL,     -- публичный ключ Base64/DER
  signature TEXT,              -- только SPK (подпись IK_Sign Боба)
  created_at INTEGER NOT NULL
)
- IK в связке (relay отдаёт IK+SPK+OPK разом, Алисе нужен IK_B для DH2).
- key_id возвращается Алисе для OPK -> она шлёт его в первом сообщении ->
  Боб находит OPK_priv по key_id.

### API relay (два новых типа сообщений)
prekeys_upload {senderId, keys:[{type,id,value,sig?}]}:
  relay сохраняет в таблицу. НЕ проверяет подписи (глупая почта).
  Митигация: НЕ логировать "user X uploaded" на диск.
prekeys_request {targetId}:
  - SPK: SELECT один, НЕ удалять (переиспользуется).
  - OPK: SELECT один + сразу DELETE (атомарно, one-time!). Если нет -> null.
  - IK: SELECT (стабильный, не удалять).
  - ответ prekeys_response: {ik, spk:{value,sig}, opk:{id,value}|null}
  - обрабатывается НА ЛЕТУ, НЕ через queue (это запрос данных, не сообщение).

### Связь с Р4.2 (graceful degradation)
opk=null в ответе -> Алиса переходит на DH1-DH3 (без DH4).
Система не падает, снижает уровень (PFS one-time -> обычный FS) ради связи.

### ОСТАЛОСЬ в Р5 (НЕ кодить relay пока — допроектировать)
- Р5.3: формат ПЕРВОГО сообщения (initial message): EK_A_pub, opk_id,
  IK_A_pub, шифртекст. Что открыто (маршрутизация), что зашифровано.
  Связь с конвертом K4 {type,to,payload}.
- Р6: обработка первого сообщения на стороне Боба (повтор DH -> SK).
Код relay+клиента — ПОСЛЕ полной спеки (не по кускам).

## РАЗВИЛКА 5.3: формат первого сообщения (Initial Packet) — ЗАФИКСИРОВАНО
Трёхслойный пакет. Граница видимости relay проходит после Слоя 2.

### Слой 1 — транспортный заголовок (виден relay, для маршрутизации)
- recipient_id (кому — Боб)
- message_type = INITIAL_HANDSHAKE (relay знает, как обрабатывать)

### Слой 2 — X3DH-заголовок (ОТКРЫТЫЙ, виден relay — это НОРМА)
- IK_A_pub (долгосрочный ключ Алисы)
- EK_A_pub (эфемерный ключ сессии)
- opk_id (какой OPK взят, или null -> graceful degradation DH1-DH3)
- ПОЧЕМУ ОТКРЫТ: Боб вычисляет SK ИЗ этих данных. Нельзя зашифровать то,
  что нужно для вычисления ключа шифрования (курица-яйцо). Это публичные
  ключи+id — их знание не даёт атакующему ничего.
- БЕЗ ПОДПИСИ Алисы: аутентификация встроена в DH-математику (DH1=IK_A↔SPK_B
  доказывает владение IK_A_priv). Успешная расшифровка Слоя 3 САМА доказывает
  подлинность Алисы. Отдельная подпись избыточна И сломала бы DENIABILITY
  (без подписи Боб не докажет третьим лицам авторство — ценно для
  авторитарного контекста).

### Слой 3 — payload (ЗАШИФРОВАН ключом SK, только Боб)
- cipher_text = AES-GCM(SK-производный Kenc, сообщение)
- nonce (для AES-GCM)
- tag (GCM целостность+аутентичность)
- REPLAY-защита: встроена в one-time OPK (повтор перехваченного пакета не
  расшифруется — OPK израсходован, Боб удалил OPK_priv). Отдельный
  транспортный nonce НЕ нужен (дублирование).

### Восстановление сессии Бобом
1. читает Слой 2 (открытый) -> IK_A, EK_A, находит OPK_priv по opk_id.
2. повторяет 4 DH (или 3 если opk_id=null) -> SK -> Kenc/Kauth (HKDF).
3. расшифровывает Слой 3. Успех = Алиса подлинна (deniable auth).

### Связь с конвертом K4
Существующий конверт {type,to,payload}: Слой 1 = type+to,
Слои 2+3 упакованы в payload. INITIAL_HANDSHAKE — новый type рядом с "msg".

## РАЗВИЛКА 6: обработка первого сообщения Бобом — ЗАФИКСИРОВАНО
Зеркало действий Алисы. Замыкает спецификацию X3DH.

### Шаг 1: читает открытый Слой 2
IK_A_pub, EK_A_pub, opk_id. Находит свои приватные: SPK_priv (всегда есть),
OPK_priv по opk_id (если ≠ null), IK_DH_priv.

### Шаг 2: повторяет 4 DH (свои приватные ↔ публичные Алисы)
DH коммутативен -> тот же секрет что у Алисы:
- DH1 = SPK_B_priv ↔ IK_A_pub
- DH2 = IK_B_priv ↔ EK_A_pub
- DH3 = SPK_B_priv ↔ EK_A_pub
- DH4 = OPK_B_priv ↔ EK_A_pub   (если opk_id ≠ null)
Порядок в IKM тот же: F ‖ DH1 ‖ DH2 ‖ DH3 ‖ DH4.

### Шаг 3: graceful degradation
opk_id = null -> IKM без DH4 (F‖DH1‖DH2‖DH3). Боб узнаёт про отсутствие DH4
ИЗ opk_id в заголовке (Р4.2: не флаг в IKM). Обе стороны детерминированно
собирают IKM одной длины -> Ambiguity-защита замкнута.

### Шаг 4: HKDF
SK = HKDF(IKM) -> Kenc, Kauth (Развилка 3).

### Шаг 5: расшифровка + ОБЯЗАТЕЛЬНАЯ проверка tag (безопасность!)
AES-GCM(Kenc) расшифровывает Слой 3. GCM-tag проверяется:
- tag сошёлся -> SK верный -> отправитель ВЛАДЕЛ IK_A_priv -> Алиса подлинна.
- tag НЕ сошёлся -> неверный SK -> подмена/повреждение -> ОТВЕРГНУТЬ.
Встроенная аутентификация: неудача расшифровки = неаутентичный отправитель.
Проверка tag ОБЯЗАТЕЛЬНА, не опциональна.

### Шаг 6: финализация (Forward Secrecy — порядок критичен!)
ТОЛЬКО ПОСЛЕ успешной расшифровки:
- удалить OPK_priv по opk_id из БД (one-time израсходован = залог FS).
- сохранить производные Kenc/Kauth (или ratchet-состояние), НЕ master secret.
- уничтожить промежуточные DH-секреты из памяти.
ПОРЯДОК: сначала успешная расшифровка -> потом удаление OPK_priv. Если
удалить до — повреждённое/атакующее сообщение лишит Боба ключа для
легитимного повтора.

## РЕАЛИЗАЦИЯ: формат объекта подписи SPK (P2) — ЗАФИКСИРОВАНО
Точная бинарная раскладка. Обе стороны собирают ОДИНАКОВЫЕ байты,
иначе verify_signature не сойдётся.

объект подписи = SPK_pub (91 байт, DER X.509)
               ‖ timestamp (8 байт, big-endian, Unix millis)
               ‖ key_id    (4 байта, big-endian, int32)
ИТОГО ровно 103 байта. Фиксированные длины -> нет неоднозначности.
Big-endian (сетевой порядок).

Боб: собирает 103 байта -> signData() (KeyStore/TEE) -> DER-подпись (~71-72б).
Алиса: получает с relay {spk_pub, timestamp, key_id, signature} ->
собирает ТЕ ЖЕ 103 байта -> verify_signature(IK_B_sign_pub, 103б, sig).

### Параметры (решено)
- Пачка OPK: 50 штук (баланс БД vs новые собеседники до пополнения).
- Ротация SPK: ОТЛОЖЕНА. Генерим один SPK при инициализации.
  Ротация (раз в неделю/месяц, как Signal) — отдельная задача. TODO.

## P4: честная модель угроз (реализация публикации prekeys)
ДАЁМ:
- E2EE: relay технически не может прочитать содержимое.
- Key Isolation: приватные ключи только на устройстве. Изъятие сервера
  не вскрывает переписку (там только публичные ключи).
- Forward Secrecy на уровне СЕССИИ (X3DH эфемерные + OPK one-time).
  Per-message FS — только с Double Ratchet (отложен).
- OPK one-time: атомарное удаление на relay (DELETE...RETURNING).

НЕ ДАЁМ (честно, чтобы не подвергать пользователя ложной уверенности):
- НЕ скрываем факт использования: DPI/провайдер видят TLS-соединение
  с известным адресом relay. Приложение видно.
- НЕ скрываем граф "пользователь <-> relay": видно, кто обращается
  к инфраструктуре. В авторитарном контексте = риск идентификации.
- НЕ скрываем метаданные соединения (время, длительность, объём).
- Маскировка от DPI (обфускация, onion, domain fronting) = ОТДЕЛЬНАЯ
  будущая задача (RETICULUM_FUTURE), НЕ входит в P4.

## УТОЧНЕНИЕ Развилки 1 (реализация identity как пары) — перед SessionManager
Обнаружено при дизайне SessionManager: IK для DH ещё не создавался.
bootstrap делал только SPK+OPK. "IK" на relay = Sign-ключ (не умеет ECDH).

### Identity = ДВЕ функциональные сущности
- IK_SIGN (в KeyStore/TEE, PURPOSE_SIGN): подписывает SPK. Приватная часть
  неизвлекаема. Из него же stableId. Алиса проверяет им подпись SPK Боба.
- IK_DH (в SQLCipher, ECDH-пара P-256): участвует в DH1, DH2. Приватная
  часть в зашифрованной БД (KeyStore не даёт sign-ключ для ECDH).

### На relay публикуем ОБА (иначе Алиса не сможет И проверить, И посчитать DH)
- ik_dh:   публичный ECDH-ключ (для DH1, DH2)
- ik_sign: публичный ECDSA-ключ (для проверки подписи SPK)
- spk + подпись, opk пачка

### DH-формулы (из работающего test_x3dh_symmetry.cpp — НЕ менять!)
DH1 = IK_A_dh ↔ SPK_B    (identity Алисы <-> signed prekey Боба)
DH2 = EK_A    ↔ IK_B_dh  (эфемерный Алисы <-> identity Боба)
DH3 = EK_A    ↔ SPK_B
DH4 = EK_A    ↔ OPK_B
ВНИМАНИЕ: DH1 = IK_A ↔ SPK_B (НЕ IK_A ↔ IK_B!). Частая ошибка.

### План правок до SessionManager
1. bootstrap: генерить IK_DH (ECDH), приватную в SQLCipher (keyType="IK_DH").
2. buildUploadJson: публиковать ik_dh И ik_sign отдельно.
3. relay: принимать оба типа.
4. JNI deriveX3DHKeys(peer_ik_dh, peer_spk, peer_opk, our_ik_dh_priv, our_ek_priv).

## РЕШЕНИЕ: Cloudflare/CDN против DDoS — ОТКЛОНЕНО
Обсуждали защиту от распределённого DDoS через Cloudflare. ОТКЛОНЕНО:
- Cloudflare терминирует TLS -> видит IP всех пользователей и метаданные.
  Это выносит граф третьей стороне (US-компания, судебные запросы) —
  ПРЯМО против Blind Delivery, которую строили.
- Преждевременно: нет пользователей/нагрузки/мотивированного противника.
ПРИНЯТЫЙ ПОДХОД: доступность < конфиденциальность. Если сервер упадёт от
DDoS — данные НЕ текут (E2E payload шифрован, метаданных нет, ключи на
устройствах). Сервер поднимется, пользователи продолжат. Временная
недоступность приемлема, компрометация — нет.
DDoS вернуться к нему с реальными пользователями (не Cloudflare, а
self-hosted/правильная юрисдикция).
