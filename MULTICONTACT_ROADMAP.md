# Многоконтактность + офлайн-доставка — план (РАЗВЕДАНО, до кода)

> Принцип: разведка -> проверка -> код. Не ломать работающее.
> Разведка (01.07.2026) вскрыла: клиентская модель НЕ готова к
> многоконтактности и офлайн-доставке. Порядок скорректирован.

## ЧТО ПОКАЗАЛА РАЗВЕДКА
- MessageEntity: text, isOwn, timestamp, status(SENT/DELIVERED/READ/PENDING)
  -> НЕТ привязки к контакту. Все сообщения в общей куче = один диалог.
- MessageDao: только insert/getAll/clearAll. Нет выборки по контакту,
  обновления статуса, удаления одного. Заготовка, не рабочий слой.
- status предусмотрен, но реально не используется (всегда дефолт SENT).
- Контакты (ContactEntity): стабильный contactId XXXX-XXXX-XXXX-XXXX,
  = SHA-256 от identity-ключа. Адресная книга на сервер НЕ уходит.
- БД: SQLCipher, fallbackToDestructiveMigration (dev).

## ВЫВОД: серверную очередь НЕЛЬЗЯ строить первой
Офлайн-доставка требует, чтобы клиент умел разложить пришедшие
сообщения по диалогам и обновить статусы. Сейчас не умеет.
Сначала клиентский фундамент, потом серверная очередь.

## ПОРЯДОК КИРПИЧЕЙ
1. МОДЕЛЬ: MessageEntity + peerId (String, стабильный ID собеседника).
   Версия БД 2->3. Destructive migration сотрёт тестовые сообщения (ок, dev).
   ВАЖНО: crypto-shredding (wipeKey) НЕ трогаем — другой механизм, фича цела.
2. DAO: выборка по peerId (открыть диалог), обновление статуса
   (PENDING->DELIVERED), удаление диалога.
3. ЛОГИКА: привязать отправку/приём к peerId в MainActivity.
   Входящее с relay -> определить отправителя -> в правильный диалог.
4. UI: список диалогов -> тап -> экран переписки с контактом.
5. СЕРВЕРНАЯ ОЧЕРЕДЬ (офлайн-доставка) — на готовом фундаменте:
   - расширяем СУЩЕСТВУЮЩИЙ Node.js relay (НЕ переписываем язык!)
   - better-sqlite3, таблица: to(peerId), payload(E2E-блоб), ttl_expiry
   - офлайн-получатель -> в очередь; подключился -> отдать -> удалить
   - TTL 7 дней (агрессивно), удаление после доставки на устройство
   - relay видит граф (from->to), НЕ видит контент. Onion — далёкий горизонт.
   - НЕ хранить payload_hash (лишний метаданный след)

## АРХИТЕКТУРНЫЕ РЕШЕНИЯ (согласовано)
- Модель А: один централизованный relay. Федерация — горизонт, НЕ сейчас.
- Очередь в SQLite (переживает ребут), НЕ бэкапится (эфемерна по TTL).
- Защита контента = E2E payload клиентом (первично).
  Шифрование БД на сервере = слабый слой от холодного диска, не переоценивать.

## ЗАМЕТКИ НА БУДУЩЕЕ (деликатность к хранилищу)
- ПЕРЕД РЕЛИЗОМ: заменить fallbackToDestructiveMigration на настоящие
  Migration(N, N+1). Иначе каждое обновление со сменой схемы сотрёт
  переписку реальных пользователей. Сейчас (dev) destructive — ок.
- crypto-shredding (wipeKey) — killer-фича, трогать только осознанно.


## НАЙДЕНО ПРИ ТЕСТЕ КИРПИЧА 3 (03.07.2026) — исправить
### БАГ A (КРИТИЧНЫЙ): stableId меняется при переустановке приложения
- Наблюдение: ID второго телефона был 1E64-A284-4BCD-50E1,
  после переустановки APK стал 139D-E0AC-BEE8-39B8.
- Причина: identity-ключ (IDENTITY_ALIAS в KeyStore) пересоздаётся
  при переустановке -> SHA-256 от него -> другой stableId.
- ПОЧЕМУ КРИТИЧНО: контакты привязаны к ID. Если ID "уплывает" при
  переустановке — вся адресная книга ломается, собеседники теряют друг друга.
- stableId ДОЛЖЕН быть по-настоящему стабильным (переживать переустановку).
- Направление: хранить identity-ключ так, чтобы переживал переустановку
  (или предупреждать, что переустановка = смена личности). Требует разведки:
  почему KeyStore-ключ не переживает — это переустановка стирает, или
  destructive migration, или что-то ещё. ФАКТ ПЕРЕД ФИКСОМ.

### БАГ B (мелкий): эхо своего pubkey от relay
- Relay присылает клиенту его СОБСТВЕННЫЙ pubkey обратно.
- В логе: peerId=свой_id перед реальным peerId собеседника.
- Итог сейчас правильный (реальный перезаписывает), но грязно.
- ФИКС: не присваивать currentPeerId, если senderId == myStableId.
