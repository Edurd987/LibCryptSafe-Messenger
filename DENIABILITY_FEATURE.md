# Дениабельность (Plausible Deniability) — приоритетная фича

> Статус: ИДЕЯ, ВЫСОКИЙ ПРИОРИТЕТ.
> ТРАНСПОРТ ГОТОВ (wss VPS relay развёрнут, см. INFRASTRUCTURE.md).
> Можно проектировать после теста связи между сетями.

## ЧТО ЭТО
Защита не только от перехвата, но и от ПРИНУЖДЕНИЯ отдать пароль.
Два входа в приложение:
- РЕАЛЬНЫЙ PIN/биометрия -> настоящий профиль с зашифрованными чатами.
- ПОДСТАВНОЙ (duress) PIN -> безобидный профиль-обманка (пустые/фейковые чаты).
Под принуждением пользователь вводит duress-PIN: атакующий видит
"чистое" приложение, реальные данные остаются скрыты и неотличимы.

## ПОЧЕМУ ЭТО KILLER-FEATURE ДЛЯ НАШЕГО ПРОЕКТА
- Мейнстрим (Signal/WhatsApp/Telegram) этого НЕ делает.
- Развивает то, что уже построено: crypto-shredding + BiometricPrompt app-lock.
- Точно ложится в позиционирование "мессенджер для тех, кому реально
  нужна защита" — фича, ради которой выбирают именно этот продукт.

## ОПОРА НА УЖЕ ГОТОВОЕ
- BiometricPrompt app-lock (есть) -> точка входа для двух PIN.
- Crypto-shredding wipe (есть) -> механика уничтожения ключей.
- SQLCipher + ключ в Keystore (есть) -> два набора ключей/БД.

## ТЕХНИЧЕСКАЯ СУТЬ (эскиз, проектировать отдельно)
- два отдельных ключа шифрования БД (реальный / duress), оба в Keystore.
- PIN/биометрия выбирает, какой ключ разблокируется.
- duress-профиль = отдельная зашифрованная БД с безобидным содержимым.
- КРИТИЧНО: реальный и подставной разделы должны быть
  криптографически НЕОТЛИЧИМЫ (нельзя по размеру/метаданным понять,
  что есть скрытый раздел). Это сложная часть — нужен аудит.
- НЕ хранить признак "есть второй профиль" нигде в открытом виде.

## ЧЕСТНЫЕ РИСКИ (проектировать осторожно)
- дениабельность легко сделать НЕНАСТОЯЩЕЙ: если по артефактам на диске
  видно, что есть скрытый раздел — защита бесполезна. Нужен аудит.
- большой attack surface, тонкая крипта. По принципу проекта
  "больше кода = больше дыр" — делать медленно, с проверкой.
- НЕ обещать пользователю больше, чем реально гарантирует реализация.

## ПОРЯДОК
1) сначала рабочий транспорт (текущий блокер всего).
2) затем проектирование дениабельности (отдельная сессия, эскиз -> аудит).
3) реализация по кирпичам с тестами.
