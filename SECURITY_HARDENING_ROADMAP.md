# Безопасность: Hardening — дорожная карта

## ПРИНЦИП
Честность и прозрачность. Защищаем максимум что возможно на
уровне приложения, и ЧЕСТНО говорим пользователю где граница.
НЕ обещаем защиту от того что приложение не может (root-зловред,
Pegasus — они выше песочницы).

## 1. ТРАНСПОРТ: wss:// + выключить cleartext
Сейчас: usesCleartextTraffic="true", транспорт ws:// (открыт)
Содержимое E2EE-зашифровано, но транспорт открыт -> метаданные
видны, возможна возня на транспортном уровне.
Делаем (при VPS-этапе):
- перейти на wss:// (TLS) через VPS + Let's Encrypt
- в манифесте usesCleartextTraffic="false"
- (опционально) certificate pinning от MITM

## 2. ЦЕЛОСТНОСТЬ APK [ВЫПОЛНЕНО]
Сейчас: нет проверки подписи -> подмену .apk не заметим
РЕАЛИЗОВАНО (EnvironmentSecurity.kt, commit 0a54bfc):
- getSignatureSha256 через GET_SIGNING_CERTIFICATES (API 28+)
- isIntegrityOk: сравнение с эталоном DEBUG_SIGNATURE_SHA256
- статус в хабе 'Ещё' (integrity_ok / integrity_fail)
ЧЕСТНАЯ ГРАНИЦА: на root обходится (Frida-хуки, патчинг байткода).
Отсекает ленивую переподписку скриптом, поднимает планку. Не абсолют.
RELEASE-ТРЕБОВАНИЕ (обязательно перед публикацией): разделить
debug/release хеши через BuildConfig.DEBUG (архитектура двух ключей).
Debug-хеш НЕ должен попасть в release APK — иначе атакующий
переподпишет стандартным debug-ключом и обойдёт проверку.

РЕАЛЬНЫЕ ПРОВЕРКИ ДЕТЕКТОРА СРЕДЫ (п.3, по факту кода):
- ROOT: 8 su-путей + Build.TAGS test-keys + 3 Magisk-пути
- ЭМУЛЯТОР: Build.FINGERPRINT/MODEL/PRODUCT/HARDWARE (goldfish/ranchu)
- ОТЛАДЧИК: Debug.isDebuggerConnected + waitingForDebugger
(НЕ проверяется: Xposed, прямой ro.kernel.qemu — не приписывать)

## 3. ДЕТЕКТОР НЕБЕЗОПАСНОЙ СРЕДЫ [ВЫПОЛНЕНО] (НЕ "сканер шпионов"!)
ВАЖНО: приложение в песочнице НЕ МОЖЕТ найти зловреда/эксплойт
(он прячется, он выше нас). Это миф, обещать НЕЛЬЗЯ.
Можем проверять КОСВЕННЫЕ признаки компрометации:
- Root detection (RootBeer): файлы su, Magisk/SuperSU,
  запись в системные пути, test-keys (кастомная прошивка)
- Эмулятор detection (признак реверса/анализа)
- Debugger detection (кто-то ковыряет приложение)
- USB-debugging включён?
Реакция: ЧЕСТНОЕ предупреждение, НЕ блокировка намертво:
  "Обнаружен root-доступ. На таком устройстве защита может быть
   обойдена. Используйте на чистом устройстве."
НЕ говорим "просканировано, шпионов нет, вы в безопасности" - ЛОЖЬ.
РЕАЛИЗОВАНО: EnvironmentSecurity.kt (свои проверки, без библиотек),
статус в хабе 'Ещё'. Ловит ОБЫЧНЫЙ root/эмулятор/отладчик.
ЧЕСТНАЯ ГРАНИЦА: продвинутое сокрытие (Zygisk DenyList) может
обойти детект. SetEdit != root (детектор точен, не ложно тревожит).
Индикатор, не доказательство (commit 08604f6).

## 4. FLAG_SECURE [ВЫПОЛНЕНО]
Запрет скриншотов/записи экрана на уровне SurfaceFlinger.
Ловит трояны-скриншотеры БЕЗ root. НЕ спасает от root/Pegasus.
Реализовано: тумблер в хабе 'Ещё', secure by default, честный
баннер при отключении. Проверено на устройстве (commit 28c6a63).

## ЧЕСТНАЯ ГРАНИЦА (что НЕ закрываем — и говорим прямо)
- Root-зловред / Pegasus: читают память ДО шифрования, выше
  песочницы. НЕ спасает НИКАКОЙ мессенджер (ни Signal, ни мы).
- Изоляция уровня Apple (Secure Enclave) = уровень ЖЕЛЕЗА+ОС,
  недостижимо для приложения поверх чужой Android.

## МОДЕЛЬ УГРОЗ (от кого защищаем)
Провайдер/DPI читает -> E2EE (есть)
Кража телефона, доступ к файлам -> SQLCipher (есть)
Другое приложение лезет в данные -> песочница Android (есть)
MITM на транспорте -> wss://TLS (план, п.1)
Подмена приложения -> APK integrity (ВЫПОЛНЕНО, п.2)
Скриншот-троян -> FLAG_SECURE (ВЫПОЛНЕНО, п.4)
Небезопасная среда -> детектор + предупреждение (ВЫПОЛНЕНО, п.3)
Root-зловред на устройстве -> НЕ защищает никто (честно)

## ГОРИЗОНТ: будущие меры (ТОЛЬКО после базового hardening п.1-3 + аудита!)
ВАЖНО: не браться пока не закрыты п.1-3 и не проверен
существующий код (крипто, логи, обработка ошибок). Больше
фич != безопаснее. Дырявая фича хуже отсутствующей.

### Forward Secrecy / Double Ratchet (как Signal)
Сейчас: разовый ECDH -> один ключ на сессию. Украли ключ ->
вся переписка читаема.
Цель: ключ меняется КАЖДОЕ сообщение (ratchet/храповик):
- symmetric ratchet (KDF-цепочка на сообщение)
- DH ratchet (периодический новый обмен)
- Forward Secrecy: старые сообщения не расшифровать при утечке
- Post-Compromise Security: восстановление после компрометации
КРИТИЧНО: НЕ писать с нуля! Использовать libsignal/libsodium.
"Don't roll your own crypto". Большая тема ~ как всё крипто-ядро.

### Прочие будущие меры
- Сверка отпечатков (safety numbers) для защиты от MITM
- PIN/биометрия на вход + авто-блокировка по таймауту
- Исчезающие сообщения (auto-delete по таймеру)
- Скрытие содержимого уведомлений на локскрине
- (опц.) Tor/onion для скрытия IP

### Важнее любых новых фич (фундамент):
- Аудит существующей крипто (нет ли ошибки в ECDH/GCM)
- Проверка что ключи/текст НЕ попадают в логи (logcat)
- Корректная обработка ошибок расшифровки (без утечек)
- В идеале — внешний аудит крипто-кода

## АУДИТ ГЕНЕРАЦИИ КЛЮЧЕЙ (проверено)
Проверено по коду MessengerApp/KeyExchange.hpp:
- ECDH на P-256 (NID_X9_62_prime256v1), EVP_PKEY_keygen OpenSSL
- ключ генерируется случайно каждую сессию, НЕ зашит
- по сети только публичный ключ, секрет через ECDH локально
- nonce через RAND_bytes, есть session_confirmation (HMAC)
ВЫВОД: генерация ключей корректна. Ручной ввод ключей НЕ нужен
(ECDH лучше). При будущем Double Ratchet рассмотреть X25519
вместо P-256 (как Signal) — но P-256 безопасна, срочности нет.

## МОДЕЛЬ УГРОЗЫ: ПРИЛОЖЕНИЕ-СОСЕД (легальный шпион)
Сценарий: на устройстве стоит ОБЫЧНОЕ приложение (не root, не
Pegasus) которое лезет в данные/телеметрию других приложений
через свои разрешения (пример класса угроз — навязанные
гос-приложения со сбором данных).

ЗАЩИЩЕНО (песочница Android + наши слои):
- Чтение нашей БД соседом -> UID-изоляция Linux + SQLCipher = НЕ может
  (файлы /data/data/com.libcryptsafe доступны только нашему UID)
- Влезть в нашу память -> изоляция процессов ядром = НЕ может
- Украсть ключ -> AndroidKeyStore в TEE (аппаратно) = НЕ может
- Снять экран -> FLAG_SECURE (если включён) = НЕ может
- Подтверждение из логов: "AppsFilter BLOCKED com.libcryptsafe ->
  com.kursx.smartbook" — Android реально изолирует приложения

ЛАЗЕЙКИ (легальные API, которыми злоупотребляют):
- Accessibility-разрешение -> сосед может ЧИТАТЬ ЭКРАН
  ЗАЩИТА: FLAG_SECURE блокирует (у нас есть)
- NotificationListener -> сосед видит текст уведомлений
  ЗАЩИТА (горизонт): не показывать текст сообщений в уведомлениях
- Метаданные системы (факт запуска, объём трафика, активность)
  -> видны, неустранимо на уровне приложения

ЧЕСТНАЯ ГРАНИЦА:
- Если пользователь САМ выдал шпиону Accessibility/root — он сам
  открыл дверь. Детектор среды ловит root, но злоупотребление
  Accessibility — это осознанный выбор пользователя.
- Метаданные факта использования на уровне ОС — не скрыть.

ВЫВОД: песочница + SQLCipher + Keystore + FLAG_SECURE закрывают
ГЛАВНОЕ (данные, ключи, память, экран). Остаётся: скрытие текста
уведомлений (горизонт) + неустранимые метаданные факта работы.

## ФОНОВЫЕ УВЕДОМЛЕНИЯ (пробел функционала, на будущее)
СЕЙЧАС: уведомлений о новых сообщениях НЕТ. WebSocket живёт
ВНУТРИ MainActivity (connectWebSocket в onCreate/onResume,
shutdown при уходе) -> при сворачивании сокет умирает ->
сообщения в фоне не приходят.

ЧТОБЫ СДЕЛАТЬ (большой заход, НЕ быстро):
1. вынести WebSocket из Activity в FOREGROUND SERVICE
   (держит сокет живым в фоне, обязательное foreground-уведомление)
2. локальное уведомление о новом сообщении С ЗАЩИТОЙ:
   текст "Новое сообщение" БЕЗ содержимого (против соседа-шпиона
   с NotificationListener) + VISIBILITY_PRIVATE
3. разрешение POST_NOTIFICATIONS (Android 13+)
4. показывать только когда app свёрнут (lifecycle-проверка)
5. звук/вибрация через NotificationChannel

РИСК: трогает рабочий сетевой слой -> делать ОСТОРОЖНО, с бэкапом,
по шагам, проверяя что переписка не сломалась. Отдельный заход
со свежими силами, НЕ под конец сессии.

ПЛЮС текущего отсутствия: нет утечки текста на локскрин
(нечего перехватить соседу). Приватность > удобство пока.
