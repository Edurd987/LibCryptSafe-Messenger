# Reticulum / LXMF — гибридный транспорт (ГОРИЗОНТ, не сейчас)

> Статус: ИДЕЯ НА БУДУЩЕЕ. НЕ начинать, пока не закрыты:
> (1) рабочий wss:// транспорт (end-to-end между сетями, подтверждён),
> (2) релиз в Google Play,
> (3) аудит существующей крипто.
> Это далёкий горизонт, сравнимый по объёму со вторым сетевым стеком.

## ЗАЧЕМ
Дать мессенджеру "неубиваемость" — связь даже без глобального интернета:
через LoRa/RNode, локальный Wi-Fi, Bluetooth, mesh-цепочки. Сценарий:
интернет отключён в регионе, но устройства всё равно общаются.

## КЛЮЧЕВОЙ ПРИНЦИП (почему идея чистая)
Это ОБЁРТКА НА УРОВЕНЬ ВЫШЕ, а не переписывание низкоуровневого стека.
Существующий код (UI, БД, крипто-ядро) НЕ трогается. Reticulum
добавляется как параллельный транспорт за абстракцией. "Больше кода =
больше дыр" — поэтому именно модульно, изолированно, и только потом.

## АРХИТЕКТУРА (как встроить изящно)

### 1. Абстракция транспорта (ценно САМО ПО СЕБЕ, до всякого Reticulum)
Сейчас сеть прибита к OkHttp WebSocket в MainActivity. Выделить интерфейс:
  interface INetworkTransport { send/receive/connect/... }
Реализации:
  - StandardIPTransport  — текущий WebSocket-relay (есть интернет)
  - ReticulumTransport   — mesh/LoRa (автономный/скрытый режим)
ПРИМЕЧАНИЕ: этот рефакторинг полезен НЕЗАВИСИМО от Reticulum —
он развязывает сетевой код и нужен в любом случае.

### 2. Сопоставление ID
Reticulum-адрес = хэш публичного ключа (Curve25519/Ed25519). Это ложится
на нашу модель: у нас уже есть стабильный ID от хэша EC-ключа в Keystore.
Destination собеседника = его адрес в mesh.

### 3. Локальный демон + IPC (чистый путь)
Референсный Reticulum/LXMF — на Python. НЕ тащить Python в бинарник.
Запускать рядом лёгкий процесс (rnsd), общаться через локальный IPC
(loopback TCP/UDP, Unix socket, Named Pipe). Мессенджер говорит демону:
"отправь эти байты на Destination 8b4a...", демон берёт на себя
маршрутизацию/шифрование/нарезку/LoRa.
ГРАБЛИ: запуск и удержание Python-демона на Android нетривиальны
(Termux/Chaquopy/foreground service) — отдельная большая задача.

### 4. Автопереключение (failover, незаметно для пользователя)
  1. Пробуем основной сервер (WebSocket/wss).
  2. Нет связи, но есть mesh/LoRa -> ReticulumTransport.
  3. Сообщение в LXMF-конверт (store-and-forward), уходит в эфир.
LXMF = официальный протокол сообщений Reticulum, store-and-forward —
идеально для оффлайна (собеседник может быть не в сети).

## ЧЕСТНЫЕ ОГОВОРКИ (почему НЕ сейчас)
- Объём огромный: фактически второй сетевой стек. Месяцы, не вечера.
- Базовый wss ещё НЕ дотестирован end-to-end. Сначала простой случай.
- LoRa/mesh — нишевый сценарий. Главное для релиза — приватность через
  обычный интернет, а не связь без интернета.
- Большой новый attack surface. По принципу проекта — только после
  фундамента и аудита.

Сохранено, чтобы не потерять. Не начинать раньше срока.
