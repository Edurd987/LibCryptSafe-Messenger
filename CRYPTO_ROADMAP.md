# Крипто-роадмап — очередь крупных задач (приоритеты)

> Записано после того, как К6 упёрся в крипто-ограничение:
> офлайн-доставка невозможна без prekeys. Обходы (QR и пр.) отвергнуты
> как тупиковые — не решают проблему, ломают UX.

## ОЧЕРЕДЬ (строгий порядок по зависимостям)

### 1. X3DH / prekeys — офлайн-доставка (ГЛАВНЫЙ КРИПТО-ПРИОРИТЕТ)
Зачем: доставка тому, кто офлайн / кого не видел давно / после перезапуска app.
Текущее ядро: эфемерный keypair (generateKeypair 1 раз за запуск app),
shared key ломается при перезапуске -> надёжного офлайна НЕТ.
Что нужно (X3DH, как в Signal):
- иерархия ключей: identity key (долгосрочный) + signed prekey +
  пачка one-time prekeys
- публикация prekeys на relay (клиент генерит, подписывает, заливает;
  relay хранит и раздаёт по одному)
- 3-4 DH-операции при установке сессии вместо одного computeSharedKey
- изменения в C++ ядре (CryptoManager): новые external-функции
- хранение prekeys на клиенте и relay (новые таблицы)
- (опционально позже) Double Ratchet поверх — forward secrecy на сообщение
ВАЖНО: самодельная крипта опасна. Развилка (НЕ решена):
готовая аудированная библиотека (libsignal) vs свой с нуля vs сначала дизайн.
Relay-очередь К6 УЖЕ ГОТОВА (SQLite+TTL) — ждёт prekeys.

### 2. АУДИТ на проникновение (Kali Linux) — ПОСЛЕ X3DH
Пентест крипто-ядра через виртуальную Kali.
ПОЧЕМУ ПОСЛЕ: аудировать надо ГОТОВОЕ стабильное ядро, а не черновик.
Ломать до X3DH = ломать то, что всё равно перепишем.
По принципу проекта (дениабельность): "делать только по-настоящему,
с аудитом, иначе вредна" — аудит обязателен после крипто-работы.

### 3. ОНЛАЙН-НАРДЫ (GAME_MOVE через сеть) — независимый довесок
Технически НЕ зависит от X3DH (нарды идут через тот же E2E-конверт К4,
sendGameEvent уже есть). Можно сделать на текущем транспорте.
Приоритет НИЖЕ крипто-ядра (нарды — фича, мессенджер — суть).
Брать, когда захочется переключиться с крипты.

## ПРИНЦИП
Не прыгать между задачами. X3DH -> аудит -> потом довески.
Крипту делать медленно, с проверкой, возможно на аудированной библиотеке.

## X3DH — РЕШЕНИЕ ПРИНЯТО: Путь 1 (свой на P-256)
Разведка ядра завершена, маяки расставлены (07.07.2026).

### ФАКТЫ О ЯДРЕ (где что лежит)
- jni_bridge.cpp (android/app/src/main/cpp/) — JNI-мост, вызывает CryptoEngine
- crypto_engine.hpp (LibCryptSafe/include/) — фасад: generate_keypair,
  compute_shared_key, encrypt/decrypt (AES-GCM), get_fingerprint,
  compute_confirmation. Session хранит aes_key + kex + handshake_done.
- KeyExchange.hpp (MessengerApp/) — ДВИЖОК DH. Кривая P-256
  (NID_X9_62_prime256v1), ключи DER 91б, KDF=SHA256(secret)->32б AES-256,
  OPENSSL_cleanse, TOFU fingerprint, HMAC session confirmation.
- CMakeLists подключает также Cryptor.cpp, local_mesh/direct_p2p/tor_tunnel
  (TransportLayer) — статус НЕ проверен (заглушки?), НЕ трогаем пока.

### ПОЧЕМУ Путь 1 (свой X3DH на P-256), НЕ libsignal
- ядро уже на P-256, написано грамотно (RAII, cleanse, DER).
- libsignal требует X25519 -> пришлось бы ЛОМАТЬ рабочий фундамент.
- X3DH математически работает на P-256 (это DH-операции на кривой).
- независимость от внешних зависимостей, прозрачность (важно для модели угроз).
- надстройка поверх готового compute_shared_key, а не переписывание.

### АРХИТЕКТУРА (намечена, детали — в дизайне)
- KeyExchange -> низкоуровневый "движок DH" (не меняем логику)
- новый SessionManager -> оркестрация X3DH поверх движка
- хранение ключей (identity, signed prekey, one-time prekeys) в SQLCipher

### СЛЕДУЮЩИЙ ШАГ (обязательно ДО кода!)
ДЕТАЛЬНЫЙ ДИЗАЙН X3DH НА БУМАГЕ:
- набор и ПОРЯДОК DH-операций (DH1..DH4) — свериться с эталоном Signal
- KDF (HKDF над конкатенацией секретов, правильный info)
- подпись signed prekey ключом identity (защита от MITM подмены prekey)
- protocol с relay: публикация/раздача prekeys, исчерпание OPK
- схема таблиц в SQLCipher
- associated data, replay-защита
НЕ КОДИТЬ до готовой спеки. Крипто-ошибка = уязвимость, не баг.
Потом: реализация по спеке -> аудит через Kali.

## АРХИТЕКТУРНЫЙ ПРИНЦIP: слои не знают друг о друге
Вопрос "заложить ли в ядро чат+каналы+нарды" — разобран. Ответ: НЕТ, не всё в ядро.

### Разделение слоёв (крипто-ядро НЕ знает типов контента)
- Крипто-ядро шифрует ПРОИЗВОЛЬНЫЕ байты (не знает: чат/ход/новость).
- Тип контента — ВНУТРИ payload (wrapper {v,type,...}), уже реализовано.
- Новый тип контента (NEWS и пр.) = новый type в wrapper, ядро НЕ трогаем.

### Что покрывает X3DH автоматически
- ЧАТ (диалог двух) — целевой кейс.
- ОНЛАЙН-НАРДЫ — та же ПАРНАЯ сессия (sendGameEvent в тот же конверт).
  X3DH для чата автоматически защищает нарды. Отдельно закладывать НЕ надо.

### Что НЕ класть в X3DH (ловушка переусложнения)
- КАНАЛЫ (один-ко-многим) — это ГРУППОВАЯ криптография (sender keys),
  ДРУГОЙ протокол, не X3DH. Класть в ядро сейчас — вредно (усложнит X3DH,
  риск сломать оба). YAGNI. Делать ОТДЕЛЬНЫМ слоем (GroupSessionManager)
  ПОТОМ, когда X3DH обкатан. См. CHANNELS_FUTURE.md.

### Единственная предусмотрительность при дизайне X3DH (бесплатная)
- SessionManager делать абстрактным: "ПАРНАЯ сессия между двумя peer",
  НЕ "чат-специфичным". Тогда: чат+нарды используют парную сессию,
  каналы (потом) = отдельный групповой слой, не конфликтует.
- Не хардкодить "chat" в имена/логику ядра.

ИТОГ слоёв: крипто-ядро (байты) <- парная сессия X3DH (чат+нарды)
<- групповая крипта (каналы, отдельно потом). Добавление каналов
НЕ тронет X3DH. Это и есть "не переписывать повторно".

## X3DH: ДВА ЭТАПА (PQC-цель сохранена, но после классики)
Уточнение порядка. Конечная цель — ПОСТКВАНТОВЫЙ X3DH (как в PQC_FUTURE.md).
Достигается ДВУМЯ проверяемыми этапами, НЕ одним прыжком.

Контекст: транспорт УЖЕ постквантовый (relay TLS 1.3 + X25519MLKEM768).
E2E-ЯДРО пока классическое (ECDH P-256) — его PQC-апгрейд = этап 2.

### ЭТАП 1: X3DH на классическом P-256
- структура асинхронного обмена: identity + signed prekey + one-time prekeys
- forward secrecy, офлайн-доставка (разблокирует relay-очередь К6)
- проверить, проаудировать через Kali
- ЭТО фундамент. Пока строим только его.

### ЭТАП 2: PQC-гибрид ПОВЕРХ готового X3DH
- в КАЖДУЮ DH-операцию добавить Kyber/ML-KEM encapsulation
- склейка секретов: ECDH_secret || Kyber_secret -> KDF (гибрид)
- опыт Kyber есть (CovertVPN, liboqs)
- это НАДСТРОЙКА на готовую структуру, а не переплетение

### ПОЧЕМУ ИМЕННО ТАК (не вместе)
PQC надстраивается на структуру X3DH (замена/дополнение примитива внутри).
Делать вместе = проектировать сложную структуру И гибридную математику разом;
при ошибке (в крипте ломается) не понять, где — в логике X3DH или в PQC.
Сначала структура правильная и проверенная -> потом броня. Один кирпич за раз.
Конечная точка ТА ЖЕ (постквантовый X3DH), путь — два проверяемых шага.
