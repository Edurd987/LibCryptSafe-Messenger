# Исходящий офлайн-буфер (доставка при возврате сети) — будущая фича
> Статус: ИДЕЯ, записано. Средний/крупный кирпич, НЕ часть текущего.
> Ценность: динамические белые списки в репрессивных сетях — интернет не
> выключают полностью, а сужают до whitelist разрешённых сервисов. Сеть
> формально есть, трафик пользователя блокируется. Фича даёт УВЕРЕННОСТЬ:
> отправил -> доставится, когда окно откроется.

## ТЕКУЩАЯ ГРАНИЦА
- Отправитель В СЕТИ + получатель офлайн -> РАБОТАЕТ (очередь relay, буфер на сервере).
- Отправитель ОФЛАЙН + закрыл приложение -> сообщение застряло в локальной БД
  отправителя, не улетит даже при возврате сети (доотправка работает только
  пока приложение открыто и держит сокет).

## ЦЕЛЬ
Исходящий буфер: офлайн-сообщение доставляется автоматически при появлении
сети, в идеале даже при ЗАКРЫТОМ приложении.

## УРОВНИ РЕАЛИЗАЦИИ
- L1 (средний блок): исходящая очередь в Room + слушатель ConnectivityManager
  -> доотправка при возврате сети, ПОКА приложение открыто.
- L2 (сложнее, отдельная задача): доотправка при ЗАКРЫТОМ приложении.
  НЕ через сокет (сокет умирает с приложением!), а через WorkManager с
  constraint NetworkType.CONNECTED -> переживает закрытие/перезагрузку,
  Android будит задачу при появлении сети. Нюансы: Doze-задержки, фоновые лимиты.

## МЕТАДАННЫЕ (решить при проектировании)
1. СНАЧАЛА ПРОВЕРИТЬ ФАКТОМ: может ли буфер слать тот же СЛЕПОЙ конверт
   (recipient + payload, без sender), что и онлайн. Если да -> ОДИН режим,
   компромисс не нужен, модель проще и безопаснее.
2. Если нет -> двухрежимный подход: буфер может раскрыть чуть больше
   метаданных, ОНЛАЙН-режим остаётся минимальным (Blind Delivery).
   Не жертвовать основным режимом ради краевого случая.
3. ОТМЕТИТЬ: фоновый трафик WorkManager сам по себе паттерн — взвесить
   при анализе трафика для модели "против репрессий".

## ПРИНЦИП (маяк)
Два режима = два набора гарантий, которые надо держать и тестировать.
Простота — тоже безопасность. Предпочесть ОДИН слепой режим, если факт
не докажет, что буфер иначе не работает.
